Customer Churn Prediction — Telco Customer Churn ML Project

End-to-end pipeline report covering data preparation, model training, evaluation, tuning, and interpretation.
Dataset: Telco Customer Churn (WA_Fn-UseC_-Telco-Customer-Churn.csv)
Target: Churn (Yes/No → 1/0)
Team: Keshav Gupta, Mudassir Iftikhar

1. Project overview

The goal of this project is to predict whether a customer will churn using historical customer information. The workflow is organized as modular stages so that each step can be understood, inspected, and reproduced.

Pipeline stages implemented

  • Load raw data and inspect structure
  • Preprocess (type fixes, missing handling, target encoding, column dropping)
  • Exploratory Data Analysis (EDA plots)
  • Feature engineering (feature split, preprocessing transformer, scaling/imputation, optional SMOTE)
  • Train multiple models
  • Evaluate models (ROC curves, confusion matrices, metrics summary)
  • Hyperparameter tuning for selected models
  • Feature importance and SHAP-based interpretation for tree models

Models used in the project

  • Logistic Regression
  • Decision Tree
  • Random Forest
  • XGBoost
  • LightGBM

XGBoost/LightGBM are included when their libraries are available; the code handles missing dependencies gracefully.

  • Missing values: No null values appeared directly in the CSV read, but TotalCharges required cleaning because blanks become NaN only after numeric coercion.

3. Preprocessing (cleaning & target encoding)

Preprocessing operations

  1. TotalCharges numeric fix:
    • The project coerces TotalCharges to numeric using errors="coerce", which converts whitespace/invalid entries into NaN.
  2. Missing handling:
    • For numeric-coerced columns (including TotalCharges), NaN values are imputed with the median.
  3. Target encoding:
    • Churn is converted from "Yes"/"No" to binary 1/0.
  4. Drop unnecessary column:
    • customerID is removed because it is an identifier and not a predictive feature.

5. Feature engineering & model-ready dataset

How features were prepared

  1. Split into X and y:
    • X = all columns except Churn
    • y = Churn (0/1)
  2. Train-test split:
    • Test size: 20%
    • Stratified by churn label to keep churn distribution consistent across splits
  3. Preprocessing transformer using ColumnTransformer:
    • Numeric columns:
      • Median imputation
      • Standard scaling
    • Categorical columns:
      • Most-frequent imputation
      • Ordinal encoding with unknown-value handling
  4. Optional SMOTE:
    • Because churn is imbalanced, SMOTE is applied only to the training set to balance the minority class.
    • This improves model learning on churn patterns without contaminating test distribution.

Feature names

For interpretation, the project tracks the original feature columns (numeric + categorical) prior to transformation. These names are used later when generating feature importance and SHAP plots.

8. Hyperparameter tuning (GridSearchCV)

After the initial training and evaluation, the project performs hyperparameter tuning for: Random Forest, XGBoost, and LightGBM. The tuning uses grid search with stratified cross-validation (ROC-AUC scoring).

Tuning configuration

  • Cross-validation folds: 5
  • Scoring metric: roc_auc
  • Refit enabled (best model refit on full training data)
  • Best estimators saved as *_tuned.joblib

Selected tuning grids (from config)

  • RandomForest: n_estimators {100,200,300}, max_depth {None,10,20}, min_samples_split {2,5,10}
  • XGBoost: n_estimators {100,200}, max_depth {3,5,7}, learning_rate {0.05,0.1,0.2}, subsample {0.8,1.0}
  • LightGBM: n_estimators {100,200}, max_depth {5,10,-1}, learning_rate {0.05,0.1}, num_leaves {31,63}

Note: Tuning is optional in the sense that the pipeline proceeds to evaluation and interpretation for the trained (non-tuned) models as well. If tuned models exist, the pipeline evaluates them and ranks by ROC-AUC.
